### Трек 1. Энергетика: Балансировка умной электросети (Smart Grid)

Контекст: Вы разрабатываете бэкенд для региональной электросети. Это IoT-система: 500 000 умных счетчиков и десятки ветрогенераторов непрерывно (раз в секунду) шлют телеметрию. Параллельно система обрабатывает текстовые журналы аварийных бригад.

- Этап 1 (Дни 1-2). Хранилище телеметрии (LSM-Tree)

    Инженерная задача (Storage Engineer): Реализовать MemTable для приема сырых метрик в RAM. При достижении лимита (например, 10 МБ) данные сортируются по sensor_id и сбрасываются на диск как иммутабельный SSTable. B-Tree здесь не справится с Extreme Write-Heavy нагрузкой.

    Данные: Напишите Python-скрипт генератора. Он должен бесконечно писать в CSV-файл строки [timestamp, sensor_id, voltage, current]. Сгенерируйте минимум 5-10 ГБ данных для тестов.

- Этап 2 (День 3). Индексация аварий и Command Bus

    Инженерная задача (Platform Engineer): Ремонтные бригады оставляют unstructured-отчеты. Реализуйте Inverted Index, чтобы по слову «замыкание» диспетчер за O(1) получал ID отчетов. Вторая задача — Append-only лог для управляющих команд («отключить фидер №5»). Счетчики должны читать этот лог по своему offset.

- Этап 3 (День 4). Биллинг и Пиковые нагрузки (MapReduce)

    Инженерная задача (Data Engineer): Ваш коллега сгенерировал 10 ГБ телеметрии. В RAM они не влезут. Напишите MapReduce, который разделит файл на чанки. Mapper выдает парные ключи (sensor_id, Wh). Reducer суммирует энергопотребление за месяц для выставления счетов клиентам.

- Этап 4 (Дни 5-6). System Design и CAP-анализ

    Бизнес-задача (Команда): Защита архитектуры. Электросеть требует AP (Availability + Partition Tolerance) дизайна. Если район теряет связь с ЦОД (Partition), счетчики не должны «падать». Обоснуйте в документе, как локальные брокеры будут буферизировать данные (Eventual Consistency) до восстановления сети.
